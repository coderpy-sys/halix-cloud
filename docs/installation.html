Getting started

Halix Cloud is a self-hosted VPS control plane (NestJS API, Next.js panel, PostgreSQL, Redis, BullMQ workers). This guide mirrors the shape of classic panel install docs (for example Pterodactyl Panel — Getting Started) but targets this repository’s stack — not Wings, not PHP, not game eggs.

Expectations. You need a Linux host (or WSL2 for dev), Docker or local Postgres/Redis, and comfort with terminals. Halix is under active development: treat production installs as “you operate the stack.”

Picking a server OS

Primary targets (same class as common panel docs):

OSVersionNotes
Ubuntu22.04 / 24.04Recommended for servers; matches docs/DEPLOYMENT.md.
Debian12+Fine with Docker Engine + Compose plugin.
Rocky / Alma8 / 9Use Docker or install Node 20+ from NodeSource or similar.
Windows10 / 11Local dev only: Node + Docker Desktop or WSL2.

OpenVZ-only VPS hosts often break Docker — prefer KVM VMs or bare metal for production.

Dependencies

  • Node.js ≥ 18.18 (LTS 20.x recommended for Next.js 15 + tooling).
  • npm 9+ (workspace install at repo root).
  • PostgreSQL 16+ (or use the provided Docker Compose service).
  • Redis 7+ (queues; same Compose file).
  • Git — clone your Halix fork or this monorepo.
  • Optional: Docker + Compose v2 for docker/docker-compose.yml.

Example: Ubuntu 24.04 (Node + Docker)

# Node 20 (example using NodeSource — verify current instructions on nodejs.org)
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs

# Docker (official docs) — then compose plugin
sudo apt-get install -y docker.io docker-compose-v2

Obtain the code

git clone <your-halix-repo-url> halix-cloud
cd halix-cloud

Repository layout (short): apps/web (Next.js), apps/api (NestJS + Prisma), docker/, docs/.

Automated installer (recommended)

Halix ships an interactive installer (Bash + optional PowerShell) that can clone from a .git URL, write Docker and app .env files with custom ports, start Postgres/Redis, run prisma db push, create the first SUPERADMIN user, and optionally register a Proxmox VE node — then emit systemd and nginx snippets under install/output/.

  • Full reference: install/README.md in the repository (same tree as this HTML).
  • From an existing clone: chmod +x install/halix-install.sh && ./install/halix-install.sh --from-repo
  • Fresh clone: set HALIX_REPO_URL (must end with .git) at the top of the script or export it, then bash install/halix-install.sh.
  • Windows: .\install\halix-install.ps1 -FromRepo (Docker Desktop + Node required).
npm run install:halix   # runs bash install/halix-install.sh (Unix / WSL)

Database & Redis

Option A — Docker (quickest)

npm run docker:up
# or: docker compose -f docker/docker-compose.yml up -d

Optional docker/.env — copy docker/.env.example to set HALIX_PG_PORT, HALIX_REDIS_PORT, and DB password for Compose variable substitution.

Defaults in Compose: user/db halix, password halix_dev_change_me unless overridden.

Option B — Managed Postgres + Redis

Create a database and user; allow TLS if your provider requires it. Set DATABASE_URL and REDIS_URL in apps/api/.env.

Environment files

cp apps/api/.env.example apps/api/.env
cp apps/web/.env.example apps/web/.env

Edit apps/api/.env (minimum):

  • DATABASE_URL — Prisma connection string.
  • REDIS_URL — e.g. redis://127.0.0.1:6379.
  • JWT_ACCESS_SECRET / JWT_REFRESH_SECRET — long random strings.
  • CORS_ORIGIN — e.g. http://localhost:3000 for dev.

Edit apps/web/.env:

  • NEXT_PUBLIC_API_URL — API origin (e.g. http://localhost:4000).
  • NEXT_PUBLIC_SOCKET_URL — usually same host as API for Socket.IO.

Install packages

From the repository root:

npm install

Database migrations

Prisma lives under apps/api. First-time schema push:

cd apps/api
npx prisma migrate dev --name init
npx prisma generate

Production deploys should use:

npx prisma migrate deploy

Run (development)

Terminal 1 — API (port 4000 by default):

npm run dev:api

Terminal 2 — Web panel (port 3000):

npm run dev
  • Panel: http://localhost:3000
  • API health: http://localhost:4000/health
  • OpenAPI UI: http://localhost:4000/docs

First user & auth

Register via API (or add a seed script later):

curl -s -X POST http://localhost:4000/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@example.com","password":"YourLongPassword12","firstName":"Admin"}'

Then login:

curl -s -X POST http://localhost:4000/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@example.com","password":"YourLongPassword12"}'

In the browser panel, set the access token for API calls (until the login UI wires automatically):

localStorage.setItem('halix_access_token', '<paste accessToken>')

Queue workers

Analogous to Pterodactyl’s queue listeners: long-running jobs (provision, reinstall, metrics) should not block HTTP requests.

  • Redis must be running.
  • Implement apps/worker (recommended) as a separate Node process that subscribes to BullMQ queues registered in the API (e.g. vps-provision).
  • Use systemd or supervisor to keep workers alive; restart on failure.
Scheduler. For periodic metrics scrape or billing reconciliation, use node-cron inside a small worker container or system cron calling a CLI — see docs/ARCHITECTURE.md roadmap.

Production paths

  1. Build: npm run build at root (builds workspaces that define the script).
  2. API: cd apps/api && npm run start:prod (or Docker image via docker/Dockerfile.api).
  3. Web: cd apps/web && npm run start (behind TLS terminator).
  4. Workers: separate service(s) with same env as API for DB/Redis.

See docs/DEPLOYMENT.md for Cloudflare, TLS, and hardening bullets.

Docker image (API)

From repo root (adjust tag/registry):

docker build -f docker/Dockerfile.api -t halix-api:local .

Ensure runtime env includes DATABASE_URL, secrets, and that migrations ran before traffic.

Reverse proxy

Put NGINX, Caddy, or Traefik in front:

  • Route / → Next.js upstream.
  • Route /v1 (and /docs if desired) → NestJS upstream.
  • WebSocket upgrade for Socket.IO path /realtime (namespace) — confirm your gateway path in Nest config.

Updating

git pull
npm install
cd apps/api && npx prisma migrate deploy
npm run build

Restart API, web, and workers. Read migration notes if Prisma schema changed.

Troubleshooting

SymptomCheck
Prisma cannot connectDATABASE_URL, firewall, Postgres listening, Docker health.
API starts but queues hangREDIS_URL, Redis maxmemory/eviction, worker process running.
Panel “Not authenticated” toastsLogin; set halix_access_token; CORS_ORIGIN must include panel URL.
Web build ESLint errorsNode 20+ recommended; run npm run lint in apps/web.
Secrets. Never commit .env files. Rotate JWT and DB passwords if exposed. Proxmox tokens belong in the DB encrypted — see architecture doc.
